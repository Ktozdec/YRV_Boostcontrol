# Программная база K-Line для ESP_Boost

Цель доработки: добавить диагностику Daihatsu YRV по `K-Line` через `L9637D`, не удаляя и не ломая текущую логику бустконтроллера. ESP32 не хранит полную PID-базу и не расшифровывает всю диагностику. PID-база, формулы и будущие PID АКПП живут в Android-приложении.

## 1. Роль ESP32

ESP32 выполняет только транспортные задачи:

- слушает `K-Line` в режиме сниффера
- принимает от телефона HEX-команду
- отправляет эту команду в `K-Line`
- ловит ответ
- возвращает сырые HEX-данные в телефон

Вся расшифровка PID, формулы, подписи и единицы измерения остаются на стороне приложения.

## 2. Режим сниффера

На первом этапе прошивка работает в режиме `listen-only`:

- `Serial2` запускается на `10400 baud`, формат `8N1`
- `GPIO16 / RX2` принимает данные с `LV3 (LLC)`
- `GPIO17 / TX2` удерживается в состоянии UART idle/high, чтобы `L9637D` отпускал шину
- `Serial2.write()` для `K-Line` на этом этапе не используется
- байты группируются в кадры по паузе `25 ms`
- последний кадр и счетчики отправляются отдельным BLE-пакетом `K`

BLE-пакет диагностики K-Line:

```json
{"K":1,"kb":120,"kf":4,"kl":11,"ko":0,"kh":"C2 F1 33 41 0C 1A F8 00"}
```

Поля:

- `kb` - сколько байт принято с момента запуска
- `kf` - сколько кадров собрано по паузам
- `kl` - длина последнего кадра
- `ko` - количество переполнений буфера кадра
- `kh` - последний кадр в hex

Этот режим нужен для проверки железа: питание `L9637D`, подтяжка `510 Ом`, защитный резистор `100 Ом`, направление `RX/TX`, стабильность `10400 baud`.

## 3. Ручной мост запроса

Телефон отправляет BLE-команду:

```text
KREQ:8310F0210C01
```

ESP32 возвращает:

```json
{"KRES":1,"ok":true,"txl":6,"rxl":8,"ov":0,"rx":"..."}
```

Этот режим нужен для ручной проверки рабочих команд и дальнейшей расшифровки PID.

## 4. Следующий этап: активное чтение KWP2000

После подтверждения, что пассивный прием видит реальные кадры, добавляем активный транспорт:

- `5-baud init` через временное переключение `GPIO17` из UART TX в GPIO output
- адрес инициализации: `0x33`
- после init возврат `Serial2` на `10400 8N1`
- ожидание key bytes от ЭБУ
- проверка checksum `modulo 256`
- фильтрация эха собственных байтов, потому что `K-Line` полудуплексная
- автомат состояния: `Idle -> Init -> Session -> RequestPid -> WaitResponse -> Decode -> Cooldown`

На этом этапе активные запросы должны быть включаемыми флагом, чтобы при проблемах можно было быстро вернуться к пассивному режиму.

## 5. Первый набор PID для чтения

Начинаем с стандартных `Service 0x01` PID из исследования:

| Данные | PID | Байты | Формула |
| --- | --- | --- | --- |
| Обороты двигателя | `0x0C` | `A B` | `((A * 256) + B) / 4` |
| Скорость авто | `0x0D` | `A` | `A` |
| Нагрузка двигателя | `0x04` | `A` | `A * 100 / 255` |
| Температура ОЖ | `0x05` | `A` | `A - 40` |
| Температура впуска | `0x0F` | `A` | `A - 40` |
| MAP / давление во впуске | `0x0B` | `A` | `A` |
| Дроссель | `0x11` | `A` | `A * 100 / 255` |
| УОЗ | `0x0E` | `A` | `(A / 2) - 64` |
| STFT Bank 1 | `0x06` | `A` | `(A * 100 / 128) - 100` |
| LTFT Bank 1 | `0x07` | `A` | `(A * 100 / 128) - 100` |
| O2 Sensor 1 Voltage | `0x14` | `A B` | `A / 200`, `B` как STFT |

## 6. Интеграция с текущей телеметрией

Текущая прошивка уже считает:

- `boost` и `tps` по `ADS1115`
- `rpm` по входу `GPIO18`
- `speed` по `GPIO19 / PCNT`
- duty и режим защиты

K-Line сначала добавляется параллельно как диагностический источник. После проверки можно сделать выбор источника:

- `RPM`: оставить текущий вход как основной, K-Line как контроль
- `Speed`: оставить `VSS` как основной, K-Line как контроль
- `TPS`: сравнить `ADS1115` и PID `0x11`
- `MAP`: сравнить `ADS1115` и PID `0x0B`
- `ECT`, `IAT`, `Ignition`, `FuelTrim`, `O2`: добавить как новые диагностические поля

## 7. Правило безопасности

До успешного пассивного чтения и проверки осциллографом активная передача в `K-Line` не включается. Это сохраняет текущий проект рабочим и позволяет отлаживать шину без риска сорвать обмен ЭБУ или коммерческого сканера.
